Lesson: Introduction to TypeScript

TypeScript is a an open-source programming language developed and maintained by Microsoft. It is a strict syntactical superset of JavaScript and adds optional static typing to the language. This means you can write JavaScript code in TypeScript files, and TypeScript will compile it down to plain JavaScript that can run in any browser or Node.js environment.

Why Use TypeScript?

  • Static Typing: Catch errors early during development rather than at runtime. This leads to more robust and maintainable code.
  • Improved Code Quality and Understandability: Types make code easier to read and reason about, especially in large projects or teams.
  • Better Tooling: TypeScript enables rich code editing experiences, including autocompletion, refactoring, and advanced error checking in IDEs like VS Code.
  • Scalability: It helps manage large codebases by providing structure and preventing common JavaScript pitfalls.
  • Access to Modern JavaScript Features: TypeScript allows you to use the latest ECMAScript features and compiles them down to older JavaScript versions for broader compatibility.

TypeScript vs. JavaScript

Think of TypeScript as JavaScript with "superpowers." Here's a simple comparison:

JavaScript:

function greet(person, date) {
  console.log(`Hello ${person}, today is ${date}!`);
}
greet("Brendan"); // Output: Hello Brendan, today is undefined! (Error not caught)

TypeScript:

function greetTS(person: string, date: Date) {
  console.log(`Hello ${person}, today is ${date.toDateString()}!`);
}
// greetTS("Brendan"); // Error: Expected 2 arguments, but got 1.
greetTS("Brendan", new Date()); // Works as expected

TypeScript enhances JavaScript by adding types, which helps in building more reliable and scalable applications. It doesn't replace JavaScript but builds upon it.

Lesson: Setting up TypeScript Environment

To start using TypeScript, you need to install it and set up a basic project structure. TypeScript is typically installed as an npm package.

Prerequisites

  • Node.js and npm: TypeScript is distributed via npm (Node Package Manager), which comes with Node.js. Ensure you have Node.js installed. You can download it from nodejs.org.

Installing TypeScript

You can install TypeScript globally or locally in your project. For most projects, local installation is preferred as it allows different projects to use different TypeScript versions.

Global Installation:

npm install -g typescript

This makes the TypeScript compiler (tsc) available anywhere on your system.

Local Installation (Recommended):

Navigate to your project directory and run:

npm init -y  # Initializes a new npm project (creates package.json)
npm install typescript --save-dev # Installs TypeScript as a dev dependency

Your First TypeScript File

  1. Create a new file, for example, hello.ts. (Note the .ts extension).
  2. Write some TypeScript code: 
    // hello.ts
function sayHello(name: string): string {
    return `Hello, ${name}!`;
}
console.log(sayHello("TypeScript"));
  3. Compile the TypeScript file to JavaScript using the TypeScript Compiler (tsc): 
    npx tsc hello.ts  # If installed locally
# OR
tsc hello.ts      # If installed globally
    This will generate a hello.js file.
  4. Run the JavaScript file: 
    node hello.js
    You should see "Hello, TypeScript!" in your console.

For larger projects, you'll use a tsconfig.json file to manage compiler options. You can generate one using npx tsc --init. This file tells the TypeScript compiler how to compile your project (e.g., target JavaScript version, module system, strictness options). We'll cover this in a later lesson.

Lesson: Basic Types

TypeScript extends JavaScript by adding types. Let's explore the fundamental types you'll encounter.

Boolean

The simplest datatype, representing true or false values.

let isLoggedIn: boolean = false;
isLoggedIn = true;

Number

As in JavaScript, all numbers in TypeScript are floating-point values. TypeScript supports decimal, hexadecimal, binary, and octal literals.

let decimal: number = 6;
let hex: number = 0xf00d;
let binary: number = 0b1010;
let octal: number = 0o744;

String

Represents textual data, using single quotes (') or double quotes ("). Template strings (backticks `) are also supported for multi-line strings and embedded expressions.

let color: string = "blue";
color = 'red';

let fullName: string = `Bob Bobbington`;
let age: number = 37;
let sentence: string = `Hello, my name is ${fullName}.

I'll be ${age + 1} years old next month.`;

Array

TypeScript allows you to work with arrays of values. You can define array types in two ways:

  1. Using the type of the elements followed by []:
    2. let list: number[] = [1, 2, 3];
  2. Using a generic array type, Array<elemType>:
    3. let names: Array<string> = ["Alice", "Bob", "Charlie"];

Tuple

Tuples allow you to express an array with a fixed number of elements whose types are known, but need not be the same.

// Declare a tuple type
let x: [string, number];
// Initialize it
x = ["hello", 10]; // OK
// Initialize it incorrectly
// x = [10, "hello"]; // Error

Enum

Enums allow us to define a set of named constants. Using enums can make it easier to document intent, or create a set of distinct cases.

enum Color {Red, Green, Blue}
let c: Color = Color.Green; // c is 1 (by default, enums start numbering at 0)

enum HttpStatus {OK = 200, NotFound = 404, InternalServerError = 500}
let status: HttpStatus = HttpStatus.OK; // status is 200

Any

Sometimes, you might need to describe the type of variables that you do not know when you are writing an application. The any type is a powerful way to work with existing JavaScript, allowing you to gradually opt-in and opt-out of type checking.

let notSure: any = 4;
notSure = "maybe a string instead";
notSure = false; // okay, definitely a boolean

Using any defeats the purpose of TypeScript's type safety, so use it sparingly.

Unknown

Similar to any, but safer. You can't perform operations on an unknown type without first asserting or narrowing it to a more specific type.

let maybe: unknown;
maybe = "hello";
// console.log(maybe.length); // Error: Object is of type 'unknown'.
if (typeof maybe === "string") {
    console.log(maybe.length); // OK
}

Void

void is a little like the opposite of any: the absence of having any type at all. You may commonly see this as the return type of functions that do not return a value.

function warnUser(): void {
    console.log("This is my warning message");
}

Null and Undefined

In TypeScript, both undefined and null actually have their own types named undefined and null respectively. By default, null and undefined are subtypes of all other types. This means you can assign null and undefined to something like number. However, when using the --strictNullChecks flag, null and undefined are only assignable to any, unknown and their respective types.

let u: undefined = undefined;
let n: null = null;

Never

The never type represents the type of values that never occur. For instance, never is the return type for a function expression or an arrow function expression that always throws an exception or one that never returns.

function error(message: string): never {
    throw new Error(message);
}

function infiniteLoop(): never {
    while (true) {}
}

Object

object is a type that represents the non-primitive type, i.e. anything that is not number, string, boolean, bigint, symbol, null, or undefined.

declare function create(o: object | null): void;
create({ prop: 0 }); // OK
create(null); // OK
// create(42); // Error
// create("string"); // Error

Often, TypeScript can infer the type of a variable from its initial value. For example:

let message = "Hello, world!"; // TypeScript infers 'string'
// message = 10; // Error: Type 'number' is not assignable to type 'string'.

While explicit type annotations are good for clarity, type inference can make your code less verbose where types are obvious.

Lesson: Interfaces

Interfaces are a powerful way to define contracts within your code as well as contracts with code outside of your project. They are one of TypeScript's core concepts and are used to define the "shape" of objects.

Our First Interface

Let's define a simple interface that describes an object with a label property of type string.

interface LabeledValue {
  label: string;
}

function printLabel(labeledObj: LabeledValue) {
  console.log(labeledObj.label);
}

let myObj = { size: 10, label: "Size 10 Object" };
printLabel(myObj); // Works because myObj has a 'label' property of type string.

The type checker only cares that the object passed to printLabel has the required properties and that those properties are of the correct type. It doesn't matter if the object has additional properties.

Optional Properties

Not all properties of an interface may be required. You can mark properties as optional by adding a ? at the end of their name.

interface SquareConfig {
  color?: string;
  width?: number;
}

function createSquare(config: SquareConfig): { color: string; area: number } {
  let newSquare = { color: "white", area: 100 };
  if (config.color) {
    newSquare.color = config.color;
  }
  if (config.width) {
    newSquare.area = config.width * config.width;
  }
  return newSquare;
}

let mySquare1 = createSquare({ color: "black" });
let mySquare2 = createSquare({ width: 50, color: "blue" });
let mySquare3 = createSquare({});

Readonly Properties

Some properties should only be modifiable when an object is first created. You can specify this by putting readonly before the name of the property:

interface Point {
  readonly x: number;
  readonly y: number;
}

let p1: Point = { x: 10, y: 20 };
// p1.x = 5; // Error! Cannot assign to 'x' because it is a read-only property.

TypeScript also provides a ReadonlyArray<T> type that is the same as Array<T> with all mutating methods removed, so you can make sure you don’t change your arrays after creation:

let a: number[] = [1, 2, 3, 4];
let ro: ReadonlyArray<number> = a;
// ro[0] = 12; // Error!
// ro.push(5); // Error!
// ro.length = 100; // Error!
// a = ro; // Error! Assigning ReadonlyArray back to mutable array is disallowed unless you use a type assertion.
a = ro as number[]; // OK with type assertion

Excess Property Checks

When passing object literals directly to a function or assigning to a variable with an interface type, TypeScript performs "excess property checking." If an object literal has any properties that the target type doesn’t have, you’ll get an error.

// interface SquareConfig defined above
// let mySquare4 = createSquare({ colour: "red", width: 100 }); // Error: Object literal may only specify known properties, and 'colour' does not exist in type 'SquareConfig'. Did you mean to write 'color'?

There are ways around this, like type assertions or adding a string index signature if appropriate.

Function Types

Interfaces can describe function types. This is useful for defining the signature of a function.

interface SearchFunc {
  (source: string, subString: string): boolean;
}

let mySearch: SearchFunc;
mySearch = function(src, sub) { // Parameter names can be different
  let result = src.search(sub);
  return result > -1;
}
// let anotherSearch: SearchFunc = function(source: string, sub: string, extra: number) { /* Error: too many params */ }

Indexable Types

Interfaces can describe types that we can "index into" like a[10], or ageMap["daniel"]. Indexable types have an index signature that describes the types we can use to index into the object, along with the corresponding return types when indexing.

interface StringArray {
  [index: number]: string;
}
let myArray: StringArray = ["Bob", "Fred"];
let first = myArray[0]; // Bob

interface NumberDictionary {
  [index: string]: number;
  length: number;    // ok, length is a number
  // name: string;   // error, the type of 'name' is not a subtype of the numeric index type
}

Class Types (Implementing an interface)

Interfaces can be implemented by classes. This enforces that the class meets the contract defined by the interface.

interface ClockInterface {
  currentTime: Date;
  setTime(d: Date): void;
}

class Clock implements ClockInterface {
  currentTime: Date = new Date();
  setTime(d: Date) {
    this.currentTime = d;
  }
  constructor(h: number, m: number) { }
}

Extending Interfaces

Interfaces can extend other interfaces, allowing you to copy members from one interface to another, which gives you a bit of flexibility in how you separate your interfaces into reusable components.

interface Shape {
  color: string;
}

interface PenStroke {
  penWidth: number;
}

interface Square extends Shape, PenStroke { // Can extend multiple interfaces
  sideLength: number;
}

let square = {} as Square; // Type assertion for simplicity
square.color = "blue";
square.sideLength = 10;
square.penWidth = 5.0;

Hybrid Types

An interface can describe an object that acts as both a function and an object, with additional properties.

interface Counter {
  (start: number): string; // Call signature
  interval: number;
  reset(): void;
}

function getCounter(): Counter {
  let counter = function (start: number) { } as Counter;
  counter.interval = 123;
  counter.reset = function () { };
  return counter;
}

let c = getCounter();
c(10);
c.reset();
c.interval = 5.0;

Type aliases (type MyType = ...) can also be used to name object types and are similar to interfaces. However, a key difference is that interfaces are "open" and can be merged (declaration merging), while type aliases are "closed" and cannot. Generally, use interfaces for defining the shape of objects or class contracts, and type aliases for naming primitives, unions, tuples, or more complex utility types.

Lesson: Classes

TypeScript offers full support for the class keyword introduced in ES6, along with additions like member visibility and abstract classes.

Basic Class Definition

class Greeter {
  greeting: string;

  constructor(message: string) {
    this.greeting = message;
  }

  greet() {
    return "Hello, " + this.greeting;
  }
}

let greeter = new Greeter("world");
console.log(greeter.greet()); // Output: Hello, world

Inheritance

TypeScript allows you to use standard object-oriented patterns. One of the most fundamental patterns in class-based programming is being able to extend existing classes to create new ones using inheritance.

class Animal {
  name: string;
  constructor(theName: string) { this.name = theName; }
  move(distanceInMeters: number = 0) {
    console.log(`${this.name} moved ${distanceInMeters}m.`);
  }
}

class Snake extends Animal {
  constructor(name: string) { super(name); }
  move(distanceInMeters = 5) {
    console.log("Slithering...");
    super.move(distanceInMeters);
  }
}

class Horse extends Animal {
  constructor(name: string) { super(name); }
  move(distanceInMeters = 45) {
    console.log("Galloping...");
    super.move(distanceInMeters);
  }
}

let sam = new Snake("Sammy the Python");
let tom: Animal = new Horse("Tommy the Palomino"); // tom is typed as Animal but is a Horse instance

sam.move();
tom.move(34);

Public, Private, and Protected Modifiers

  • public (default): Members are accessible from anywhere.
  • private: Members are only accessible within the defining class.
  • protected: Members are accessible within the defining class and by instances of subclasses.
class Person {
  public name: string;
  private age: number;
  protected id: string;

  public constructor(name: string, age: number) {
    this.name = name;
    this.age = age;
    this.id = Math.random().toString(36).substring(7);
  }

  public getDetails() {
    return `Name: ${this.name}, Age: ${this.age}`; // Can access private 'age' here
  }
}

class Employee extends Person {
  private department: string;

  constructor(name: string, age: number, department: string) {
    super(name, age);
    this.department = department;
  }

  public getEmployeeDetails() {
    // console.log(this.age); // Error: 'age' is private to Person
    return `${super.getDetails()}, Department: ${this.department}, ID: ${this.id}`; // Can access protected 'id'
  }
}

let john = new Person("John Doe", 30);
console.log(john.name); // OK
// console.log(john.age); // Error: 'age' is private
// console.log(john.id); // Error: 'id' is protected

let jane = new Employee("Jane Smith", 28, "Engineering");
console.log(jane.getEmployeeDetails());

Readonly Modifier

You can make properties readonly by using the readonly keyword. Readonly properties must be initialized either at their declaration or in the constructor.

class Octopus {
  readonly name: string;
  readonly numberOfLegs: number = 8;

  constructor(theName: string) {
    this.name = theName;
  }
}
let dad = new Octopus("Man with the 8 strong legs");
// dad.name = "Man with the 3-piece suit"; // Error! name is readonly.

Parameter Properties

TypeScript offers a shorthand for declaring properties and initializing them from constructor parameters.

class User {
  // 'readonly email: string;' and 'this.email = email;' are done automatically
  constructor(readonly email: string, public username: string, private age: number) {}

  displayInfo() {
    console.log(`Email: ${this.email}, Username: ${this.username}, Age: (private)`);
  }
}
let user = new User("test@example.com", "tester", 25);
console.log(user.email); // test@example.com
console.log(user.username); // tester
// console.log(user.age); // Error: 'age' is private

Accessors (Getters & Setters)

TypeScript supports getters/setters as a way of intercepting accesses to an object member.

let passcode = "secret passcode";

class SecureDocument {
  private _fullName: string = "";

  get fullName(): string {
    return this._fullName;
  }

  set fullName(newName: string) {
    if (passcode && passcode === "secret passcode") {
      this._fullName = newName;
    } else {
      console.log("Error: Unauthorized update of fullName!");
    }
  }
}

let doc = new SecureDocument();
doc.fullName = "John Doe"; // Uses the setter
if (doc.fullName) { // Uses the getter
  console.log(doc.fullName);
}

Static Properties and Methods

Static members are visible on the class itself rather than on the instances. They are accessed using the class name.

class Grid {
  static origin = { x: 0, y: 0 };
  
  calculateDistanceFromOrigin(point: { x: number; y: number; }) {
    let xDist = (point.x - Grid.origin.x);
    let yDist = (point.y - Grid.origin.y);
    return Math.sqrt(xDist * xDist + yDist * yDist) / this.scale;
  }
  constructor(public scale: number) { }
}

let grid1 = new Grid(1.0);  // 1x scale
let grid2 = new Grid(5.0);  // 5x scale

console.log(Grid.origin); // { x: 0, y: 0 }
console.log(grid1.calculateDistanceFromOrigin({ x: 10, y: 10 }));
console.log(grid2.calculateDistanceFromOrigin({ x: 10, y: 10 }));

Abstract Classes

Abstract classes are base classes from which other classes may be derived. They may not be instantiated directly. Unlike an interface, an abstract class may contain implementation details for its members. The abstract keyword is used to define abstract classes as well as abstract methods within an abstract class.

abstract class Department {
  constructor(public name: string) {}

  printName(): void {
    console.log("Department name: " + this.name);
  }

  abstract printMeeting(): void; // must be implemented in derived classes
}

class AccountingDepartment extends Department {
  constructor() {
    super("Accounting and Auditing"); // constructors in derived classes must call super()
  }

  printMeeting(): void {
    console.log("The Accounting Department meets each Monday at 10am.");
  }

  generateReports(): void {
    console.log("Generating accounting reports...");
  }
}

let department: Department; // ok to create a reference to an abstract type
// department = new Department(); // error: cannot create an instance of an abstract class
department = new AccountingDepartment(); // ok to create and assign a non-abstract subclass
department.printName();
department.printMeeting();
// department.generateReports(); // error: method doesn't exist on declared type Department!
(department as AccountingDepartment).generateReports(); // OK with type assertion

Because classes create types, you can use them in the same places you would be able to use interfaces. For example, a class declaration creates a type with the same name, representing the instance side of the class.

class Point {
    x: number = 0;
    y: number = 0;
}

interface Point3d extends Point { // A class can be extended by an interface
    z: number;
}

let point3d: Point3d = {x: 1, y: 2, z: 3};

Lesson: Functions

Functions are fundamental building blocks in JavaScript and TypeScript. TypeScript adds type safety to function parameters and return values.

Function Declaration and Types

You can define types for function parameters and return values:

// Named function
function add(x: number, y: number): number {
  return x + y;
}

// Anonymous function (function expression)
let multiply = function(x: number, y: number): number {
  return x * y;
};

// Arrow function
const subtract = (x: number, y: number): number => {
  return x - y;
};

Typing the Function

We can also explicitly type the variable that holds a function expression:

let divide: (a: number, b: number) => number;

divide = (x, y) => { // Parameter names can be different
  if (y === 0) throw new Error("Cannot divide by zero");
  return x / y;
};

// console.log(divide("10", 2)); // Error: Argument of type 'string' is not assignable to parameter of type 'number'.
console.log(divide(10, 2)); // 5

The parameter types and return type together constitute the function's "type signature."

Optional and Default Parameters

In TypeScript, every parameter is assumed to be required by the function. You can mark a parameter as optional by appending a ? to its name. Default-initialized parameters are also considered optional.

// Optional parameter
function buildName(firstName: string, lastName?: string): string {
  if (lastName) {
    return firstName + " " + lastName;
  } else {
    return firstName;
  }
}
let result1 = buildName("Bob");             // Bob
let result2 = buildName("Bob", "Adams");    // Bob Adams
// let result3 = buildName("Bob", "Adams", "Sr."); // Error, expected 2 arguments, but got 3.

// Default parameter
function greetUser(name: string, greeting: string = "Hello"): string {
  return `${greeting}, ${name}!`;
}
console.log(greetUser("Alice"));          // Hello, Alice!
console.log(greetUser("Alice", "Hi"));  // Hi, Alice!

Optional parameters must follow required parameters. However, default parameters don't need to appear after all required parameters. If a default-initialized parameter comes before a required parameter, users need to explicitly pass undefined to get the default value.

Rest Parameters

Rest parameters allow a function to accept an indefinite number of arguments as an array.

function buildFullName(firstName: string, ...restOfName: string[]): string {
  return firstName + " " + restOfName.join(" ");
}

let employeeName = buildFullName("Joseph", "Samuel", "Lucas", "MacKinzie");
console.log(employeeName); // Joseph Samuel Lucas MacKinzie

The rest parameter must be the last parameter in the function signature.

this and Arrow Functions

In JavaScript, the value of this can change depending on how a function is called. Arrow functions capture the this where the function is created rather than where it is invoked. This is particularly useful for callbacks and methods that need to preserve their original this context.

interface Card {
  suit: string;
  card: number;
}
interface Deck {
  suits: string[];
  cards: number[];
  createCardPicker(this: Deck): () => Card; // Explicitly typing 'this'
}

let deck: Deck = {
  suits: ["hearts", "spades", "clubs", "diamonds"],
  cards: Array(52),
  createCardPicker: function(this: Deck) {
    return () => { // Arrow function captures 'this' from createCardPicker
      let pickedCard = Math.floor(Math.random() * 52);
      let pickedSuit = Math.floor(pickedCard / 13);
      return { suit: this.suits[pickedSuit], card: pickedCard % 13 };
    }
  }
}

let cardPicker = deck.createCardPicker();
let pickedCard = cardPicker();
console.log("card: " + pickedCard.card + " of " + pickedCard.suit);

TypeScript helps manage this by allowing you to specify the type of this in a function signature. If you specify a this parameter, it must be the first parameter, and it's only used for type checking (it's erased during compilation).

Overloads

JavaScript is inherently a very dynamic language. It’s not uncommon for a single function to return different types of objects based on the shape of the arguments passed in. TypeScript allows you to declare function overloads to document these variations.

function combine(input1: number, input2: number): number;
function combine(input1: string, input2: string): string;
function combine(input1: any, input2: any): any { // This is the implementation signature
  if (typeof input1 === 'number' && typeof input2 === 'number') {
    return input1 + input2;
  } else if (typeof input1 === 'string' && typeof input2 === 'string') {
    return input1 + input2;
  }
  throw new Error("Inputs must be both numbers or both strings.");
}

let numResult = combine(1, 2);       // 3
let strResult = combine("Hello", "World"); // HelloWorld
// let errResult = combine("Hello", 1); // Error: No overload matches this call.

The implementation signature (the one with the actual function body) must be compatible with all the overload signatures. It is not visible from the "outside". When calling an overloaded function, TypeScript will try to match the call with one of the overload signatures.

Overloads are useful when a function can behave quite differently based on its input types and you want to provide strong type safety for each case. However, if the variations are numerous or complex, using union types or generic parameters might be a more manageable approach.

Lesson: Generics

Generics allow you to write reusable code that can work with a variety of types rather than a single one. This is a core feature for building scalable and type-safe components and functions.

The Identity Function Problem

Consider a simple identity function that returns whatever is passed into it:

// Using 'any' - loses type information
function identityAny(arg: any): any {
  return arg;
}
let outputAny = identityAny("myString"); // outputAny is 'any'

// How can we make it preserve the type of the argument?

Using any is not ideal because we lose the type information. We want the function to return the specific type that was passed in.

Introducing Generics

A generic function uses a type variable, a special kind of variable that works on types rather than values. This type variable (commonly T) allows us to capture the type the user provides and use it later.

function identity<T>(arg: T): T {
  return arg;
}

// Calling the generic function
let output1 = identity<string>("myString"); // Type of output1 is 'string'
let output2 = identity<number>(100);       // Type of output2 is 'number'

// Type argument inference (often, TypeScript can infer T)
let output3 = identity("myString");        // Type of output3 is 'string'
let output4 = identity(true);              // Type of output4 is 'boolean'

The <T> after the function name declares T as a type variable. We can then use T as the type for the argument and the return value.

Working with Generic Type Variables

When you start to use generics, you might be tempted to assume that T always has certain properties. For example, if you want to log the length of arg:

// This will cause an error:
// function loggingIdentity<T>(arg: T): T {
//   console.log(arg.length); // Error: T doesn't have .length
//   return arg;
// }

To fix this, we can constrain T to types that have a length property, for example, an array of T:

function loggingIdentity<T>(arg: T[]): T[] {
  console.log(arg.length); // Array has a .length, so no more error
  return arg;
}
// Or, more generally:
function loggingIdentityWithInterface<T>(arg: { length: number }): T {
  console.log(arg.length);
  return arg; // Note: return type needs to be { length: number } too, or more specific
}
// A better way using an interface for the constraint:
interface Lengthwise {
    length: number;
}
function loggingIdentityConstrained<T extends Lengthwise>(arg: T): T {
    console.log(arg.length); // Now we know it has a .length property
    return arg;
}
loggingIdentityConstrained("hello"); // string has length
loggingIdentityConstrained([1, 2, 3]); // array has length
// loggingIdentityConstrained(10); // Error: number does not have length

Generic Types (Interfaces and Classes)

You can also create generic interfaces and classes.

Generic Interface:

interface GenericIdentityFn<T> {
  (arg: T): T;
}

function identityFromInterface<T>(arg: T): T {
  return arg;
}

let myIdentity: GenericIdentityFn<number> = identityFromInterface;
console.log(myIdentity(5)); // 5
// console.log(myIdentity("hello")); // Error: string is not assignable to number

Generic Class:

class GenericNumber<T> {
  zeroValue: T;
  add: (x: T, y: T) => T;

  constructor(zero: T, addFn: (x: T, y: T) => T) {
    this.zeroValue = zero;
    this.add = addFn;
  }
}

let myGenericNumber = new GenericNumber<number>(0, (x, y) => x + y);
console.log(myGenericNumber.add(5, 10)); // 15

let myGenericString = new GenericNumber<string>("", (x, y) => x + y);
console.log(myGenericString.add("hello", " world")); // "hello world"

Generic Constraints

We saw an example of constraints with extends Lengthwise. This ensures that the type variable T will have at least the properties of Lengthwise.

interface Vehicle {
  wheels: number;
}

function drive<T extends Vehicle>(vehicle: T) {
  console.log(`Driving a vehicle with ${vehicle.wheels} wheels.`);
}

drive({ wheels: 4 }); // OK
drive({ wheels: 2, type: "motorcycle" }); // OK, extra properties are fine
// drive({ name: "Bicycle" }); // Error: Property 'wheels' is missing

Using Type Parameters in Generic Constraints

You can declare a type parameter that is constrained by another type parameter.

function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

let x = { a: 1, b: "hello", c: true };
let aValue = getProperty(x, "a"); // number
let bValue = getProperty(x, "b"); // string
// let dValue = getProperty(x, "d"); // Error: Argument of type '"d"' is not assignable to parameter of type '"a" | "b" | "c"'.

Here, keyof T is the union of known, public property names of T. K extends keyof T ensures that key is a valid property of obj. T[K] is the type of the property key on obj.

Generics are heavily used in utility types (like Partial<T>, Readonly<T>), array methods (map<U>, filter), Promises (Promise<T>), and in frameworks like React for component props and state. They are crucial for writing flexible and robust typed code.

Lesson: Enums

Enums (enumerations) allow you to define a set of named constants. Using enums can make your code more readable and less error-prone by restricting variables to a predefined set of values.

Numeric Enums

By default, enums are number-based. They auto-increment from 0, but you can set the value of the first member or any member.

enum Direction {
  Up,    // 0
  Down,  // 1
  Left,  // 2
  Right  // 3
}
let dir: Direction = Direction.Up;
console.log(dir); // Output: 0

enum ResponseStatus {
  Success = 200,
  NotFound = 404,
  Error = 500 // Auto-increment from here if next ones are not initialized
}
let status: ResponseStatus = ResponseStatus.NotFound;
console.log(status); // Output: 404

Numeric enums also support reverse mapping from value to name:

console.log(Direction[0]); // Output: "Up"
console.log(ResponseStatus[200]); // Output: "Success"

String Enums

String enums are a similar concept, but their values are initialized with strings. String enums do not have auto-incrementing behavior and each member must be explicitly initialized with a string literal or another string enum member.

enum LogLevel {
  Info = "INFO",
  Warning = "WARN",
  Error = "ERROR",
  Debug = "DEBUG"
}

function logMessage(level: LogLevel, message: string) {
  console.log(`[${level}] ${message}`);
}

logMessage(LogLevel.Info, "Application started.");
// logMessage("OTHER", "This won't compile"); // Error

String enums offer better readability when debugging, as the string value is often more meaningful than a number. They do not have a reverse mapping from value to name.

Heterogeneous Enums

Enums can technically have a mix of string and numeric members, but this is generally not recommended due to potential confusion.

enum Mixed {
  No = 0,
  Yes = "YES",
  // Maybe = 1 // Allowed
}

Computed and Constant Members

Enum members can have values that are computed. An enum member is considered constant if:

  • It is the first member in the enum and has no initializer, in which case it’s assigned the value 0.
  • It does not have an initializer and the preceding enum member was a numeric constant. In this case the value of the current enum member will be the value of the preceding enum member plus one.
  • It is initialized with a constant enum expression (a subset of TypeScript expressions that can be fully evaluated at compile time).

In all other cases enum member is considered computed.

enum FileAccess {
  // constant members
  None, // 0
  Read    = 1 << 1, // 2 (computed from constant expression)
  Write   = 1 << 2, // 4
  ReadWrite  = Read | Write, // 6 (computed from constant expression)
  // computed member
  G = "123".length // 3
}
console.log(FileAccess.Read); // 2
console.log(FileAccess.G);    // 3

Enums at Runtime

Enums are real objects that exist at runtime. For example, the Direction enum above would compile to JavaScript similar to this:

// Generated JavaScript for Direction enum (simplified)
var Direction;
(function (Direction) {
    Direction[Direction["Up"] = 0] = "Up";
    Direction[Direction["Down"] = 1] = "Down";
    Direction[Direction["Left"] = 2] = "Left";
    Direction[Direction["Right"] = 3] = "Right";
})(Direction || (Direction = {}));

Const Enums

To avoid the overhead of generating code for an enum object and to achieve inlining of enum values, you can use const enums. Const enums are completely removed during compilation after their values are substituted.

const enum HttpStatusCodes {
  OK = 200,
  MovedPermanently = 301,
  BadRequest = 400
}

let code: HttpStatusCodes = HttpStatusCodes.OK;
// console.log(HttpStatusCodes[200]); // Error: A const enum member can only be accessed using a string literal.
// Accessing HttpStatusCodes itself after compilation would also be an error as it's erased.

The compiled JavaScript for using HttpStatusCodes.OK would directly inline the value 200.

For simple cases, especially with string values, a union of string literals can be a lightweight alternative to string enums:

type Alignment = "left" | "right" | "center";
let textAlign: Alignment = "left";
// textAlign = "justify"; // Error

// Compared to:
// enum TextAlignmentEnum { Left = "left", Right = "right", Center = "center" }
// let enumAlign: TextAlignmentEnum = TextAlignmentEnum.Left;

String literal unions don't create runtime objects and are often preferred for their simplicity when you just need a set of specific string values.

Lesson: Advanced Types

TypeScript has a rich type system that includes several advanced type constructs. These allow for more expressive and precise type definitions.

Intersection Types

An intersection type combines multiple types into one. This new type has all members of all the types in the intersection. It's like an "AND" operation for types.

interface Loggable {
  log(message: string): void;
}
interface Serializable {
  serialize(): string;
}

type LoggableSerializable = Loggable & Serializable;

class DataItem implements LoggableSerializable {
  constructor(private data: any) {}

  log(message: string): void {
    console.log(`[LOG: ${new Date().toISOString()}] ${message} - Data: ${this.data}`);
  }

  serialize(): string {
    return JSON.stringify(this.data);
  }
}

let item = new DataItem({ id: 1, value: "test" });
item.log("Processing item");
let serializedItem = item.serialize();
console.log(serializedItem);

Union Types

A union type describes a value that can be one of several types. We use the vertical bar (|) to separate each type. It's like an "OR" operation for types.

function padLeft(value: string, padding: string | number): string {
  if (typeof padding === "number") {
    return Array(padding + 1).join(" ") + value;
  }
  if (typeof padding === "string") {
    return padding + value;
  }
  throw new Error(`Expected string or number, got '${typeof padding}'.`);
}

console.log(padLeft("Hello", 4));       // "    Hello"
console.log(padLeft("Hello", "---")); // "---Hello"
// console.log(padLeft("Hello", true)); // Error

If a value has a union type, we can only access members that are common to all types in the union. To access specific members, you'll need to use type guards.

Type Guards and Differentiating Types

Type guards are expressions that perform a runtime check that guarantees the type in some scope. Common type guards include typeof, instanceof, and property checks.

interface Bird {
  fly(): void;
  layEggs(): void;
}
interface Fish {
  swim(): void;
  layEggs(): void;
}

function getPet(): Bird | Fish {
  return Math.random() > 0.5 ? { fly: () => {}, layEggs: () => {} } : { swim: () => {}, layEggs: () => {} };
}

let pet = getPet();
pet.layEggs(); // OK, common property

// Using 'in' operator as a type guard
if ("fly" in pet) {
  pet.fly(); // pet is narrowed to Bird
} else {
  pet.swim(); // pet is narrowed to Fish
}

// User-defined type guards
function isBird(p: Bird | Fish): p is Bird {
  return (p as Bird).fly !== undefined;
}

if (isBird(pet)) {
  pet.fly();
} else {
  pet.swim();
}

Type Aliases

Type aliases create a new name for a type. Type aliases can be used for primitives, unions, intersections, tuples, or any other types that you’d otherwise have to write by hand.

type Point = { x: number; y: number };
type ID = string | number;
type Tree<T> = {
  value: T;
  left?: Tree<T>;
  right?: Tree<T>;
};

let startPoint: Point = { x: 0, y: 0 };
let userId: ID = "user123";
let numberTree: Tree<number> = { value: 1, left: { value: 0 }, right: { value: 2 } };

Interfaces vs. Type Aliases: Type aliases are similar to interfaces but have some key differences. Interfaces can be merged (declaration merging), while type aliases cannot. Interfaces are generally preferred for object shapes and class contracts, while type aliases are good for naming unions, intersections, or complex utility types.

String Literal Types

String literal types allow you to specify the exact value a string must have.

type Easing = "ease-in" | "ease-out" | "ease-in-out";

function animate(mode: Easing) {
  console.log(`Animating with ${mode}`);
}
animate("ease-in");
// animate("linear"); // Error: Argument of type '"linear"' is not assignable to parameter of type 'Easing'.

Numeric Literal Types

Similar to string literal types, you can have numeric literal types.

type DiceRoll = 1 | 2 | 3 | 4 | 5 | 6;
function rollDice(): DiceRoll {
  return (Math.floor(Math.random() * 6) + 1) as DiceRoll;
}
let myRoll = rollDice();

Discriminated Unions (Tagged Unions)

A common pattern that combines union types, literal types, and type guards. Each type in the union contains a common property (the discriminant) with a literal type, allowing TypeScript to narrow down the type.

interface Square {
  kind: "square"; // Discriminant property
  size: number;
}
interface Rectangle {
  kind: "rectangle";
  width: number;
  height: number;
}
interface Circle {
  kind: "circle";
  radius: number;
}
type Shape = Square | Rectangle | Circle;

function area(s: Shape): number {
  switch (s.kind) {
    case "square": return s.size * s.size; // s is Square
    case "rectangle": return s.width * s.height; // s is Rectangle
    case "circle": return Math.PI * s.radius ** 2; // s is Circle
    default:
      const _exhaustiveCheck: never = s; // Ensures all cases are handled
      return _exhaustiveCheck;
  }
}

Index Types (keyof and Lookup Types)

keyof T produces a union of string or numeric literal types representing the public property names of T. Lookup types (T[K]) allow you to access the type of a property.

interface Person {
  name: string;
  age: number;
  location: string;
}
type PersonProps = keyof Person; // "name" | "age" | "location"

function getPropertyTyped<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}
let person: Person = { name: "Alice", age: 30, location: "Wonderland" };
let personName: string = getPropertyTyped(person, "name");
let personAge: number = getPropertyTyped(person, "age");

Mapped Types

Mapped types allow you to create new types by transforming properties from an existing type. They build on index signatures and keyof.

// Makes all properties of T optional
type Partial<T> = {
  [P in keyof T]?: T[P];
};
// Makes all properties of T readonly
type Readonly<T> = {
  readonly [P in keyof T]: T[P];
};

interface UserProfile {
  id: number;
  username: string;
  email?: string;
}
type PartialUserProfile = Partial<UserProfile>; // All props become optional
type ReadonlyUserProfile = Readonly<UserProfile>; // All props become readonly

TypeScript provides several built-in mapped types like Partial<T>, Readonly<T>, Pick<T, K>, and Record<K, T>.

Conditional Types

Conditional types allow you to choose a type based on a condition. They take the form T extends U ? X : Y.

interface IdLabel { id: number; /* ... */ }
interface NameLabel { name: string; /* ... */ }

type NameOrId<T extends number | string> = T extends number ? IdLabel : NameLabel;

function createLabel<T extends number | string>(idOrName: T): NameOrId<T> {
  throw "unimplemented";
}

let a = createLabel("typescript"); // Type is NameLabel
let b = createLabel(2.8);      // Type is IdLabel
let c = createLabel(Math.random() ? "hello" : 42); // Type is NameLabel | IdLabel

Conditional types, especially with the infer keyword, are powerful for creating complex utility types (e.g., ReturnType<T>, Parameters<T>).

The never type is useful for exhaustiveness checking in discriminated unions. If you add a new variant to the Shape union type and forget to handle it in the area function's switch statement, the default case will try to assign the unhandled shape to _exhaustiveCheck (which is of type never). This will cause a compile-time error, alerting you to the unhandled case.

Lesson: Modules & Namespaces

As applications grow, organizing code becomes crucial. TypeScript provides modules and namespaces to help structure your code, manage dependencies, and prevent naming conflicts.

Modules (ES Modules / External Modules)

TypeScript shares the same module concept as ECMAScript 2015 (ES6). Any file containing a top-level import or export is considered a module. Modules are executed within their own scope, not in the global scope. This means variables, functions, classes, etc. declared in a module are not visible outside the module unless they are explicitly exported.

Exporting from a module:

// validator.ts
export interface StringValidator {
  isAcceptable(s: string): boolean;
}

export const numberRegexp = /^[0-9]+$/;

export class ZipCodeValidator implements StringValidator {
  isAcceptable(s: string) {
    return s.length === 5 && numberRegexp.test(s);
  }
}

// Can also export separately
class LettersOnlyValidator implements StringValidator {
  isAcceptable(s: string) {
    return /^[A-Za-z]+$/.test(s);
  }
}
// export { LettersOnlyValidator }; // Single export
export { LettersOnlyValidator as MyLettersValidator }; // Renaming export
export default class DefaultValidator { /* ... */ } // Default export

Importing into a module:

// main.ts
import DefaultValidator, { ZipCodeValidator, MyLettersValidator as LetterVal, StringValidator } from "./validator";
import * as ValidatorModule from "./validator"; // Import entire module as an object

let myZipValidator = new ZipCodeValidator();
let myLetterValidator = new LetterVal();
let defaultVal = new DefaultValidator();

let strings = ["Hello", "98052", "101"];
strings.forEach(s => {
  console.log(`"${s}" ${myZipValidator.isAcceptable(s) ? " matches" : " does not match"} zip`);
});

let v = new ValidatorModule.ZipCodeValidator(); // Using alias
  • export makes declarations available to other modules.
  • import brings declarations from other modules into the current scope.
  • Modules can have one default export.
  • import * as Alias from "./module" imports all exports into Alias.

Namespaces (Internal Modules)

Namespaces are a TypeScript-specific way to organize code. They were more common before ES modules became standard. A namespace provides a way to group related code and avoid polluting the global scope. If you're working with modern JavaScript (ES6+), modules are generally preferred over namespaces for organizing code across files.

// validation.ts (Illustrative, for a single file or legacy setup)
namespace Validation {
  export interface StringValidator {
    isAcceptable(s: string): boolean;
  }

  const lettersRegexp = /^[A-Za-z]+$/;
  const numberRegexp = /^[0-9]+$/;

  export class LettersOnlyValidator implements StringValidator {
    isAcceptable(s: string) {
      return lettersRegexp.test(s);
    }
  }

  export class ZipCodeValidator implements StringValidator {
    isAcceptable(s: string) {
      return s.length === 5 && numberRegexp.test(s);
    }
  }
}

// Using the namespace
let strings2 = ["Hello", "98052", "101"];
let validators: { [s: string]: Validation.StringValidator; } = {};
validators["ZIP code"] = new Validation.ZipCodeValidator();
validators["Letters only"] = new Validation.LettersOnlyValidator();

for (let s of strings2) {
  for (let name in validators) {
    console.log(`"${s}" - ${validators[name].isAcceptable(s) ? "matches" : "does not match"} ${name}`);
  }
}

Namespaces can be nested and can span multiple files if compiled together (e.g., using --outFile with tsc, which is less common now).

To use a namespace defined in another file (when not using ES modules and compiling multiple files together):

// <!-- Assuming validation.ts and main_namespace.ts are compiled together -->
// <!-- main_namespace.ts -->
// /// <reference path="validation.ts" />
// let v = new Validation.ZipCodeValidator();

The triple-slash reference (/// <reference path="..." />) is a directive to the compiler about inter-file dependencies for namespaces.

Modules vs. Namespaces

  • Modules (ES Modules):
    • File-based: Each file is a module.
    • Standardized: Aligns with ECMAScript standard.
    • Explicit imports/exports: Clear dependency management.
    • Better for modern JavaScript development, especially with bundlers like Webpack, Rollup, or Vite.
    • Recommended for most new projects.
  • Namespaces:
    • Not file-based inherently (can be split across files if compiled together).
    • TypeScript-specific feature.
    • Good for organizing code within a single file or when targeting environments without module loader support (less common now).
    • Can help structure global variables from libraries that don't use modules.
    • Less used in modern development favouring ES Modules.

Ambient Modules and Declaration Files (.d.ts)

When working with JavaScript libraries that don't have TypeScript typings, or when describing the shape of external modules, you use declaration files (.d.ts). These files don't contain implementations, only type information.

Ambient Module Declaration:

// my-library.d.ts
declare module "my-untyped-library" {
  export function doSomething(value: string): boolean;
  export const version: string;
}

// app.ts
// import { doSomething } from "my-untyped-library";
// doSomething("test");
// This would work if "my-untyped-library" actually existed and was configured.

Many popular JavaScript libraries have community-maintained type definitions available on DefinitelyTyped (@types/package-name on npm).

For modern TypeScript projects, ES Modules are the standard and recommended way to organize code and manage dependencies between files. Namespaces are a legacy feature that can be useful in specific scenarios, particularly when dealing with older code or global script includes, but are generally avoided in new module-based projects.

Lesson: Decorators

Decorators are a special kind of declaration that can be attached to classes, methods, accessors, properties, or parameters. They provide a way to add annotations and meta-programming syntax for class declarations and members. Decorators are an experimental feature in TypeScript and require enabling the experimentalDecorators compiler option in tsconfig.json.

Decorators are functions that are called at runtime with information about the decorated declaration.

Enabling Decorators

To use decorators, you need to enable the experimentalDecorators option in your tsconfig.json:

{
  "compilerOptions": {
    "target": "ES5", // Or newer
    "experimentalDecorators": true,
    "emitDecoratorMetadata": true // Often used with reflection libraries
  }
}

Class Decorators

A class decorator is applied to the constructor of a class and can be used to observe, modify, or replace a class definition. It receives the constructor of the class as its argument.

// Decorator function
function sealed(constructor: Function) {
  Object.seal(constructor);
  Object.seal(constructor.prototype);
  console.log("Class sealed:", constructor.name);
}

@sealed // Apply the decorator
class GreeterClass {
  greeting: string;
  constructor(message: string) {
    this.greeting = message;
  }
  greet() {
    return "Hello, " + this.greeting;
  }
}

// GreeterClass is now sealed. New properties cannot be added, existing ones cannot be removed.
// GreeterClass.newProp = 1; // Would cause error in strict mode or fail silently

If a class decorator returns a value, it will replace the class declaration (must be a constructor function).

Method Decorators

A method decorator is applied to a property descriptor for a method and can be used to observe, modify, or replace a method definition. It receives three arguments:

  1. Either the constructor function of the class for a static member, or the prototype of the class for an instance member.
  2. The name of the member.
  3. The property descriptor for the member.
function enumerable(value: boolean) {
  return function (target: any, propertyKey: string, descriptor: PropertyDescriptor) {
    descriptor.enumerable = value;
    console.log(`Method ${propertyKey} enumerable set to ${value}`);
  };
}

class PointClass {
  private _x: number;
  private _y: number;
  constructor(x: number, y: number) {
    this._x = x;
    this._y = y;
  }

  @enumerable(false) // Make getCoordinates non-enumerable
  getCoordinates() {
    return `(${this._x}, ${this._y})`;
  }
}

let p = new PointClass(10, 20);
for (let key in p) {
  console.log(key); // Will not log 'getCoordinates' if enumerable is false
}

If the method decorator returns a value, it will be used as the Property Descriptor for this member.

Accessor Decorators

Accessor decorators are similar to method decorators but are applied to accessors (getters/setters). They receive the same arguments as method decorators.

function configurable(value: boolean) {
  return function (target: any, propertyKey: string, descriptor: PropertyDescriptor) {
    descriptor.configurable = value;
  };
}

class UserAccount {
  private _name: string;
  constructor(name: string) { this._name = name; }

  @configurable(false)
  get name(): string { return this._name; }
  // set name(newName: string) { this._name = newName; } // Would need its own decorator if desired
}
// Trying to delete UserAccount.prototype.name would fail if configurable is false

Property Decorators

A property decorator is declared just before a property declaration. It receives two arguments:

  1. Either the constructor function of the class for a static member, or the prototype of the class for an instance member.
  2. The name of the member.

Property decorators cannot directly modify the property's value or type, but they can be used to record metadata or perform other side effects. The return value of a property decorator is ignored.

import "reflect-metadata"; // Polyfill for Reflect API, often used with decorators

function format(formatString: string) {
  return Reflect.metadata("format", formatString);
}

function getFormat(target: any, propertyKey: string) {
  return Reflect.getMetadata("format", target, propertyKey);
}

class MyData {
  @format("Hello, %s")
  greeting: string;

  constructor(message: string) {
    this.greeting = message;
  }
}

let md = new MyData("World");
console.log(getFormat(md, "greeting")); // "Hello, %s" (requires emitDecoratorMetadata: true)

Parameter Decorators

A parameter decorator is declared just before a parameter declaration. It is applied to the function for a class constructor or method declaration. It receives three arguments:

  1. Either the constructor function of the class for a static member, or the prototype of the class for an instance member.
  2. The name of the member (method name or undefined for constructor).
  3. The ordinal index of the parameter in the function’s parameter list.

Parameter decorators can only be used to observe that a parameter has been declared on a method. The return value is ignored.

function required(target: any, propertyKey: string | symbol, parameterIndex: number) {
  console.log(`Parameter at index ${parameterIndex} of ${String(propertyKey)} is marked as required.`);
  // Typically used to store metadata about required parameters
}

class BugReport {
  type = "report";
  title: string;

  constructor(t: string) {
    this.title = t;
  }

  print(@required verbose: boolean, @required user: string) {
    if (verbose) {
      console.log(`Bug: ${this.title} by ${user}`);
    } else {
      console.log(this.title);
    }
  }
}
let br = new BugReport("Crash on load");
br.print(true, "dev_user");

Decorator Composition

Multiple decorators can be applied to a single declaration. They are evaluated in a specific order: expressions are evaluated top-to-bottom, and the resulting functions are called bottom-to-top (like function composition).

function f() {
  console.log("f(): evaluated");
  return function (target: any, propertyKey: string, descriptor: PropertyDescriptor) {
    console.log("f(): called");
  }
}

function g() {
  console.log("g(): evaluated");
  return function (target: any, propertyKey: string, descriptor: PropertyDescriptor) {
    console.log("g(): called");
  }
}

class CExample {
  @f()
  @g()
  method() {}
}
// Output order:
// f(): evaluated
// g(): evaluated
// g(): called
// f(): called

Decorators are powerful for meta-programming tasks such as:

  • Logging or instrumentation (e.g., logging method calls).
  • Modifying behavior (e.g., debouncing a method, making it non-enumerable).
  • Data validation (e.g., validating method parameters or property values).
  • Dependency injection frameworks (e.g., Angular heavily uses decorators).
  • ORM (Object-Relational Mapper) libraries (e.g., TypeORM uses decorators to define entities and columns).

While powerful, use them judiciously as they are an advanced feature and can make code harder to understand if overused or poorly implemented.

Lesson: Working with JavaScript (JS Integration)

TypeScript is a superset of JavaScript, which means all valid JavaScript code is also valid TypeScript code. This makes integrating TypeScript into existing JavaScript projects or using JavaScript libraries in TypeScript projects relatively straightforward.

Gradual Adoption

You can introduce TypeScript into an existing JavaScript project incrementally:

  1. Rename .js files to .ts: Start with a few files. TypeScript will likely report some errors, which you can fix.
  2. Set allowJs compiler option: In tsconfig.json, set "allowJs": true. This allows TypeScript files to import JavaScript files and vice-versa. The TypeScript compiler can also process .js files.
  3. Set checkJs compiler option: Optionally, set "checkJs": true in tsconfig.json. This enables type checking for JavaScript files (using JSDoc annotations for type information).
  4. Add types gradually: Start adding type annotations to your .ts files. For .js files being checked, use JSDoc comments for types.
// tsconfig.json example for mixed project
{
  "compilerOptions": {
    "outDir": "./dist",
    "allowJs": true,
    // "checkJs": true, // Optional: type-check JS files
    "target": "es5",
    "module": "commonjs", // Or "esnext" etc.
    // "noImplicitAny": true, // Good for TS files
    // "strict": true       // Stricter checks
  },
  "include": [
    "./src/**/*" // Include both .ts and .js files in src
  ]
}

Using JavaScript Libraries

When you want to use a JavaScript library (e.g., from npm) in your TypeScript project, you need type definitions for that library. This allows TypeScript to understand the library's API and provide type checking and autocompletion.

1. DefinitelyTyped (@types):

The DefinitelyTyped project is a massive repository of high-quality TypeScript type definitions for thousands of JavaScript libraries. These are typically installed as @types packages from npm.

For example, to use Lodash:

npm install lodash
npm install --save-dev @types/lodash

Then you can import and use it with type safety:

import _ from 'lodash';

let numbers = [1, 2, 3, 4, 5];
let shuffled = _.shuffle(numbers);
console.log(shuffled); // 'shuffled' is typed as number[]

2. Libraries with Bundled Types:

Many modern JavaScript libraries are now written in TypeScript or include their own type definition files (.d.ts) directly in their npm package. In such cases, you usually don't need to install a separate @types package.

3. Writing Your Own Declaration Files (.d.ts):

If a library doesn't have type definitions available, you can write your own. This involves creating a .d.ts file that describes the shape of the library's API.

For a simple module, you might create my-js-lib.d.ts:

// src/my-js-lib.d.ts
declare module 'my-js-lib' {
  export function processData(data: any): string;
  export const version: number;
}

Then, in your TypeScript code:

// import { processData, version } from 'my-js-lib'; // Assuming my-js-lib exists
// console.log(processData({ name: "Test" }));
// console.log(version);

You might need to configure typeRoots or paths in tsconfig.json if your custom .d.ts files are not automatically discovered.

Type Checking JavaScript Files with JSDoc

If you have JavaScript files and want some level of type checking without converting them to TypeScript, you can use JSDoc annotations. With checkJs: true, TypeScript will use these annotations.

// utils.js
/**
 * Adds two numbers.
 * @param {number} x First number.
 * @param {number} y Second number.
 * @returns {number} The sum of x and y.
 */
export function add(x, y) {
  return x + y;
}

/** @type {string} */
export const defaultGreeting = "Hello";

// main.ts
// import { add, defaultGreeting } from './utils.js'; // Note the .js extension might be needed depending on module config
// console.log(add(5, 10));
// console.log(defaultGreeting.toUpperCase());
// add("5", 10); // TypeScript would flag an error here if checkJs is on and it processes utils.js

Compiling JavaScript with TypeScript

When allowJs is true, TypeScript can also compile (transpile) .js files. This is useful if you want to use newer JavaScript features in your .js files and have TypeScript transpile them down to an older target (e.g., ES5), similar to Babel.

  • any type: When integrating JS, you might encounter many implicit any types. Gradually replace them with specific types or use unknown for better safety.
  • Declaration file quality: Not all @types packages are perfect. Sometimes they might be outdated or incomplete.
  • Module resolution: Ensure your tsconfig.json settings (module, moduleResolution, baseUrl, paths) are correctly configured for how your project imports modules, especially in mixed TS/JS setups.
  • Start small: When migrating a large JS project, start with a small, isolated part of the codebase.
  • Leverage editor support: Modern IDEs like VS Code provide excellent TypeScript support, which can greatly help in identifying and fixing type issues during integration.

Lesson: Configuration (tsconfig.json)

The tsconfig.json file in the root of a TypeScript project specifies the root files and the compiler options required to compile the project. It's the central place for configuring how the TypeScript compiler (tsc) behaves.

Generating a tsconfig.json

You can create a tsconfig.json file manually, or you can generate one using the TypeScript compiler:

npx tsc --init

This command creates a tsconfig.json file with sensible defaults and comments explaining various options.

Key Sections and Common Options

A tsconfig.json file typically has a compilerOptions section and can also include sections like include, exclude, and extends.

compilerOptions: This is where most configuration happens.

  • target: Specifies the ECMAScript target version for the compiled JavaScript output (e.g., "ES5", "ES2015", "ESNext"). 
    "target": "es2016"
  • module: Specifies the module system for the output (e.g., "CommonJS", "ES6", "AMD", "UMD"). 
    "module": "commonjs"
  • outDir: Redirect output structure to the specified directory. 
    "outDir": "./dist"
  • rootDir: Specifies the root directory of input files. Used to control the output directory structure with outDir. 
    "rootDir": "./src"
  • strict: Enables all strict type-checking options (noImplicitAny, noImplicitThis, strictNullChecks, strictFunctionTypes, etc.). Highly recommended. 
    "strict": true
  • noImplicitAny: Raise error on expressions and declarations with an implied any type. 
    "noImplicitAny": true
  • strictNullChecks: When true, null and undefined have their own distinct types and you’ll get a type error if you try to use them where a concrete value is expected. 
    "strictNullChecks": true
  • esModuleInterop: Enables compatibility with Babel-emitted ECMAScript modules by creating namespace objects for all imports. Allows default imports from modules with no default export. 
    "esModuleInterop": true
  • allowJs: Allow JavaScript files to be compiled. 
    "allowJs": true
  • checkJs: Report errors in .js files. Works in conjunction with allowJs. 
    "checkJs": false
  • sourceMap: Generates corresponding .map files for debugging. 
    "sourceMap": true
  • declaration: Generates corresponding .d.ts files for your library. 
    "declaration": true
  • lib: List of library files to be included in the compilation (e.g., ["ES2020", "DOM"]). If not specified, defaults based on target. 
    "lib": ["dom", "es2017"]
  • jsx: Specifies JSX code generation: "preserve", "react", "react-jsx", "react-jsxdev". 
    "jsx": "react-jsx"
  • baseUrl and paths: For configuring module path aliases. 
    "baseUrl": "./",
"paths": {
  "@components/*": ["src/components/*"],
  "@utils/*": ["src/utils/*"]
}
  • experimentalDecorators and emitDecoratorMetadata: For enabling decorator support. 
    "experimentalDecorators": true,
"emitDecoratorMetadata": true

include, exclude, and files:

These top-level properties specify which files are part of the compilation process.

  • files: An array of file paths. Only these specific files will be compiled. 
    "files": [
  "src/main.ts",
  "src/utils.ts"
]
  • include: An array of glob patterns specifying files to include. 
    "include": [
  "src/**/*" // All files in src directory and subdirectories
]
  • exclude: An array of glob patterns specifying files to exclude from compilation. Commonly used for node_modules, output directories, etc. 
    "exclude": [
  "node_modules",
  "dist",
  "**/*.spec.ts" // Exclude test files
]

If files is specified, include and exclude are ignored. If neither files nor include is specified, the compiler defaults to including all TypeScript files (*.ts, *.tsx, *.d.ts) in the containing directory and subdirectories, except those specified by exclude.

extends:

This property allows one tsconfig.json file to inherit configurations from another. This is useful for managing multiple configurations (e.g., base config, dev config, prod config) or for using shared configurations from npm packages.

// tsconfig.prod.json
{
  "extends": "./tsconfig.base.json",
  "compilerOptions": {
    "sourceMap": false,
    "removeComments": true
  }
}

Using tsconfig.json

  • Running tsc without any input files will compile the project based on the tsconfig.json in the current directory or parent directories.
  • Running tsc -p ./path/to/tsconfig.json (or --project) allows you to specify a custom path to a tsconfig file.
  • Many build tools and IDEs (like VS Code, Webpack with ts-loader, Vite) automatically recognize and use tsconfig.json.
  • Enable strict mode: This catches many common errors and leads to more robust code.
  • Keep it versioned: tsconfig.json should be committed to your version control system.
  • Understand your target environment: Set target and module appropriately for where your JavaScript will run.
  • Use include and exclude: Be explicit about what files are part of your project to avoid accidentally compiling unwanted files or missing important ones.
  • Read the official documentation: The TypeScript handbook has comprehensive documentation on all compiler options.
